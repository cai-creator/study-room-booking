### 技术栈

| 层次 | 技术 | 版本 |
|------|------|------|

| 图标 | Lucide Icons | 1.8 CDN |
| 密码加密 | BCrypt (Hutool) | — |

### 架构特点

- **无 Spring Security** — 认证授权通过自定义 `HandlerInterceptor` 实现（`JwtInterceptor` + `RoleInterceptor`）
- **无 WebSocket** — 前端通过 30 秒轮询实现数据刷新
- **三层角色** — `STUDENT` / `ADMIN` / `SUPER_ADMIN`
- **Context Path**: `/api` — 所有接口位于 `/api/*`
- **端口**: 8081
- **多环境配置**: `dev` (默认 MySQL) / `prod` (MySQL) / `h2` (H2 内存库)
- **逻辑删除**: `sys_user`、`campus`、`building`、`floor`、`study_room`、`seat`、`reservation`、`blacklist`、`no_show_record` 均使用 `deleted` 字段
- **乐观锁**: `reservation` 表使用 `version` 字段（`@Version`），由 MyBatis-Plus `OptimisticLockerInnerInterceptor` 自动处理
- **权限控制策略（2026-07-09 更新）**: `JwtInterceptor` 采用「GET 方法 + 路径前缀」细粒度放行机制——空间查询接口（`/campuses`、`/buildings`、`/floors`、`/rooms`、`/seats`）和仪表盘接口（`/dashboard/`）的 GET 请求无需认证，但 POST/PUT/DELETE/PATCH 写操作仍需 Token 校验

## 3. 后端 API 接口测试

> **通用规则**: 
> - 所有接口返回统一格式 `{ code, message, data, timestamp }`
> - `code=200` 表示成功
> - `code=401` 表示未认证
> - `code=403` 表示无权限
> - 分页接口返回 `{ list, total, pageNum, pageSize }`
> - 所有需要认证的接口需携带 Header: `Authorization: Bearer <token>`

## 5. 端到端 (E2E) 测试场景

以下为核心业务流程的端到端测试。

### 5.1 学生完整预约流程

```
1. 学生注册账号 (POST /api/auth/register) — v1.1 新增：前端支持自助注册
2. 学生登录 (POST /api/auth/login)
3. 查看校区列表 (GET /api/campuses) — 无需 Token（JWT 公共 GET 放行）
4. 按校区筛选楼栋 (GET /api/buildings?campusId=1) — 无需 Token
5. 按楼栋筛选楼层 (GET /api/floors?buildingId=1) — 无需 Token
6. 按楼层筛选自习室 (GET /api/rooms?floorId=1) — 无需 Token
7. 选择自习室 → 查看座位 (GET /api/seats?roomId=1) — 无需 Token
8. 选择座位 → 查看可用时段 (GET /api/seats/{seatId}/available-slots?date=...) — 无需 Token
9. 创建预约 (POST /api/reservations) — 需 Token
10. 在"我的预约"确认 (GET /api/reservations/my) — 需 Token
11. 按时签到 (POST /api/checkin { roomId, seatCode }) — 需 Token（v1.1: 参数改为 roomId+seatCode）
12. 暂离 (POST /api/temporary-leave { roomId, seatCode }) — 需 Token
13. 返回座位 (POST /api/return-seat { roomId, seatCode }) — 需 Token
14. 签退离开 (POST /api/checkout { roomId, seatCode }) — 需 Token
15. 查看预约历史 (GET /api/reservations/my?status=CHECKED_OUT) — 需 Token
```

### 5.2 管理员空间管理流程

```
1. 管理员登录
2. 创建校区 (POST /api/campuses)
3. 创建楼栋 (POST /api/buildings)
4. 创建楼层 (POST /api/floors)
5. 创建自习室 (POST /api/rooms)
6. 批量生成座位 (POST /api/rooms/{roomId}/seats/generate)
7. 标记特殊座位 (PATCH /api/rooms/{roomId}/seats/tags)
8. 在空间管理中验证层级结构
9. 在学生端验证新自习室可见
```

### 5.3 爽约与黑名单流程

```
1. 学生预约座位但未签到
2. 等待定时任务执行 (60s 后标记 NO_SHOW)
3. 重复 3 次（7 天内）
4. 等待定时任务执行 (300s 后自动拉黑)
5. 验证该学生无法预约 (POST /api/reservations → 5001)
6. 管理员在黑名单中查看 (GET /api/blacklist)
7. 管理员手动解除或等待到期自动解除
8. 验证学生恢复预约能力
```

### 5.4 并发预约冲突测试

```
1. 两个学生同时查询同一座位同一时段的可用性
2. 学生A先创建预约
3. 学生B尝试创建 → 应返回 RESERVATION_CONFLICT (3007)
4. 验证乐观锁：同时取消同一预约，第二次应失败或忽略
```

## 附录

### A. 已知问题与注意事项

1. **~~空间模块路径重复~~** (已在 2026-07-09 修复): `CampusController` 等 `@RequestMapping("/api/...")` 已修正为 `@RequestMapping("/...")`，与 `context-path: /api` 不再叠加。
2. **重复实体类**: `reservation` 表对应 `Booking` 和 `Reservation` 两个实体，`seat` 表对应 `Seat` 和 `SeatControl`。测试时需注意 MyBatis 别名冲突。
3. **乐观锁**: `BookingService.cancelBooking` 等操作依赖 MyBatis-Plus 的 `updateById` 自动处理 `@Version` 字段。
4. **TOCTOU 竞态**: `BookingService.createBooking` 中时间冲突检查存在注释中标注的并发窗口，高并发测试可能暴露问题。
5. **H2 兼容性**: MySQL 特有语法（如 `DATEADD` 在 H2 中不同）在 H2 profile 下可能不一致，建议定时任务测试使用 MySQL。
6. **JwtInterceptor 公共 GET 放行 (v1.1)**: 空间查询和仪表盘接口的 GET 请求无需 Token，但 POST/PUT/DELETE/PATCH 写操作仍需。空间相关测试用例在设计时需确认是否需要预先登录获取 Token。
7. **CAS 已移除 (v1.1)**: 登录页不再包含 CAS 认证 Tab，改为「学生登录」/「管理员登录」身份 Tab + 注册模式。`config.js` 中的 `casEnabled` 和 `casLoginUrl` 已删除。
8. **更新状态接口参数变更 (v1.1)**: `updateRoomStatus` 和 `updateStatus` 从前端 JSON body 改为 query parameter 传参（`?status=X`）。
9. **添加黑名单流程变更 (v1.1)**: 前端先通过 `UserAPI.getUsers` 按用户名查 userId，再调用 `SeatAPI.addToBlacklist`，需要用户存在于系统中。
10. **签到签退参数变更 (v1.1)**: 前端调用签到/签退/暂离/返回座位 API 时改为传 `roomId` + `seatCode`（而非 `reservationId`），与后端 `CheckinController` 保持一致。
